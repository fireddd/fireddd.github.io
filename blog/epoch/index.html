Epoch

phonepe · 2022–2025 · the cron for a container orchestrator

PhonePe runs its containers on Drove, a distributed container orchestrator built in-house. Epoch is the piece that schedules work onto it: a time-based container job scheduler. If Drove is the thing that runs containers, Epoch is the cron that decides when.

I owned Epoch for online merchant services — the systems behind PhonePe's merchant-facing payments.

The scripts nobody owned

A scheduler accumulates scripts. Every job that has to run nightly, hourly, or after some other job ends up living in it, and over time nobody quite owns the pile. I introduced structural reforms to how the scripts in online merchant services were written, so they stopped being one-off shell files and started having a shape.

The most valuable thing I fixed there wasn't a feature. It was the missing-logs issue — jobs would run and leave nothing useful behind, which meant that when something went wrong the answer to "what happened?" was a shrug. That was an org-wide problem, not just ours.

On top of it I wrote the reconciliation and reporting jobs: the ones that go through pending transactions, work out what actually settled, and close them out. In payments, a transaction stuck in limbo is not a cosmetic problem — it is somebody's money.

Writing the CLI

Epoch had no command-line client. I wrote epoch-cli from the first commit — it's open source and in Python.

The interesting part of a CLI for a scheduler is that it has to make a distributed system feel like a local one. What went in:

  • a topology plugin, and filtering on it, so you can see what's actually scheduled where;
  • cluster management commands, including pause-all — the command you want to exist at 3am;
  • skip and override, for the times a job should not run, or should run differently, right now;
  • logs for runs, so a failed job can be diagnosed without going to find the container it died in;
  • Poetry packaging and a developer setup, so the next person could actually contribute;
  • naming made consistent with drove-cli, because two tools for one system should not have two vocabularies.

I also stripped the PhonePe-specific constructs out of it. That's the work that turns an internal tool into something that can stand on its own — and it's why the CLI is public today.
