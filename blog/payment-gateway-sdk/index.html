PhonePe

The SDK that merchants bet their checkout on

I owned PhonePe's B2B payment gateway SDKs and wrote the Java one. Then I refused to ship it.

  • Java
  • Open source
  • API design
  • DPoP (RFC 9449)
  • Payments
  • Tens of crores in TPV

Why an SDK at all

PhonePe's B2B payment gateway onboards external merchants who put our payment APIs inside their own checkout. Before the SDK, every one of them integrated against raw REST: reading the docs, hand-rolling request signing, auth, retries, error handling.

Every merchant re-derived the same solution, and every merchant got it slightly wrong in a different way. Those mistakes did not stay theirs — an integration bug surfaces as a customer failing to pay, which is a PhonePe problem and a support ticket and a merchant who is now less sure about us. Integration friction was a direct drag on merchant activation.

So the SDK was not a convenience library. It was the golden path, made hard to leave. It's public, and tens of crores in total payment volume have gone through the SDK family.

The release I held back

We built against v1 of the payment APIs, deliberately: most merchants were on v1, and the newer path had uncertainty around it. That was the right call at the time.

Then, right at release, the v2 APIs started landing — and the SDK, as structured, did not account for the v2 contract. Business wanted it out. Fast.

A broken integration inside a public SDK is a one-way door with an external blast radius.

Here is what that means concretely. If I ship, merchants build against my structure. When v2's contracts change underneath them, their integrations break — and I cannot fix that for them, because the code is now in their repository, in production, taking real money. There is no rollback for a design that other companies have already built on.

So I said no, and restructured the SDK first so integrations would survive the v2 change.

Two things made that stick rather than making me the person blocking the release. I got the architect aligned on why the restructure was non-negotiable — the argument was the one-way door, not my preference. And I scoped it so it shipped within a week. The business timeline held. The durability was free, in the end, because I fought for it at the only moment it was still cheap.

The launch broke

At launch, our first merchants' transactions were blocked by the fraud and risk service. The cause was small and stupid, as these always are: a missing merchant-id header. Support surfaced it, and I identified and rolled a fix within a couple of days. I also folded the regulator's requested guidelines into the SDK, and traced an API-key mapping problem at the gateway that showed up when a merchant rotated their production keys.

The fix is not the interesting part. This is: I should have caught it before a merchant did.

What I should have run was a closed user group with a handful of merchants, and a dev-assisted live onboarding of the very first one, before general availability. That catches a missing header in a controlled room instead of in front of the customer you most wanted to impress. It is now how I would launch anything customer-facing, and I would rather say that plainly than pretend the launch was clean.

Weeks to a day

Merchants couldn't generate their own test credentials. Every sandbox key had to be requested from the integrations team by hand. That is a queue, and a queue between a developer and their first successful API call is where integrations go to die.

I built a self-serve toggle so a merchant could mint their own test credentials from the production dashboard. The security question is obvious — you have just let people create credentials — so it is secured with a public/private key and DPoP (RFC 9449): a short-TTL proof-of-possession token bound to the HTTP method, the URL and a timestamp, verified at the gateway before auth even runs. The merchant id is embedded in the token, so a merchant can only ever mint their own test credentials.

Landing it meant design, product, integrations, infra, appsec and UI. It is the least glamorous list in this post and it was most of the work.

Four SDKs, one behaviour

I owned the SDK offerings, not just the Java one. The Python, .NET and JavaScript SDKs were built by three or four engineers — juniors and interns — and I mentored them rather than writing the SDKs myself.

The trap there is obvious in hindsight and invisible at the time: four SDKs written by four people become four dialects. Same gateway, four different ideas about what an error means, four retry policies, four ways of naming the same field. So I gated at the design level, not the code level. Get the behaviour agreed before anyone opens an editor, and the implementations converge on their own.

Making it public

Once the Java SDK was solid, it was living in an internal repository, serving only the merchants already in our sales funnel. That felt like a waste of the thing.

A public SDK is a channel. It reaches developers who will never talk to a salesperson, and it lets a merchant's engineers read the code before they commit to the payment gateway that will handle their money — which is exactly what a good engineer does before making that decision.

So I drove the open-source release: appsec approval, the architects' approvals, and the public repository. It is on GitHub at PhonePe/phonepe-pg-sdk-java.

That is the part I am most glad about, and not for the reasons I expected. It means that when I tell you I designed a payment SDK that merchants bet their checkout on, you do not have to take my word for it. You can go and read it.
