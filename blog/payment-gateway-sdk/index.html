The SDK that merchants bet their checkout on

My first project at PhonePe: I owned the B2B payment gateway SDKs end to end, and wrote the Java one.

  • Java
  • Open source
  • API design
  • DPoP (RFC 9449)
  • Payments
  • Prior art: Stripe
  • Funnel analytics
  • Tens of crores in TPV

Why an SDK at all

PhonePe's B2B payment gateway onboards external merchants who put our payment APIs inside their own checkout. Before the SDK, every one of them integrated against raw REST: reading the docs, hand-rolling request signing, auth, retries, error handling.

Every merchant re-derived the same solution, and every merchant got it slightly wrong in a different way. Those mistakes did not stay theirs — an integration bug surfaces as a customer failing to pay, which is a PhonePe problem and a support ticket and a merchant who is now less sure about us. Integration friction was a direct drag on merchant activation.

So the SDK was not a convenience library. It was the golden path, made hard to leave. It's public, and tens of crores in total payment volume have gone through the SDK family.

My first project here

This was my first project at PhonePe. I wanted to do everything the thing needed in order to succeed — and I was given the room to, because I was handed it end to end.

Not "write the Java client". All of it:

  • Design — I designed the entire library.
  • Implementation — I wrote it.
  • Documentation — the thing a merchant's engineer actually reads first.
  • Testing.
  • CI/CD — the pipelines that build and ship it.
  • Release — including, eventually, the open-source one.

And beyond the Java one: every decision about the SDK offering went through me. What the SDKs did, how they behaved, what a merchant could expect from any of them — that was mine to get right or get wrong.

So the first thing I did was go and read somebody else's homework. Stripe and the other giants had solved this years earlier, in public. I went through their SDKs — not to copy the shape, but to work out why each thing was shaped the way it was. Why errors are modelled that way. Why the client is constructed like that. Why idempotency sits where it sits.

A payments SDK has a lot of decisions in it that only look arbitrary until you understand what they are defending against. Somebody has already paid for those lessons, and they published the answers. Not reading them would have been vanity.

That groundwork is what made the calls that followed possible. Owning the whole thing gave me the standing to make them; reading the people who had already solved it is what told me which way to go. If I had only been writing the client, the shape of the library would have been someone else's problem to weigh. It was mine.

The launch broke

At launch, our first merchants' transactions were blocked by the fraud and risk service. The cause was small and stupid, as these always are: a missing merchant-id header. Support surfaced it, and I identified and rolled a fix within a couple of days. I also folded the regulator's requested guidelines into the SDK, and traced an API-key mapping problem at the gateway that showed up when a merchant rotated their production keys.

The fix is not the interesting part. This is: I should have caught it before a merchant did.

What I should have run was a closed user group with a handful of real merchants, and a dev-assisted live onboarding of the very first one, before general availability. Real merchants are not whitelisted. They would have hit the fraud check in a controlled room, instead of in front of the customer we most wanted to impress.

The lesson I actually took is broader than the header. A test environment that is friendlier than production is not a test environment. Whitelisting made our tests pass and made them worthless, and I did not notice because passing tests feel like evidence. It is now the first thing I check: what does my test setup make impossible to see?

Weeks to a day

Merchants couldn't generate their own test credentials. Every sandbox key had to be requested from the integrations team by hand. That is a queue, and a queue between a developer and their first successful API call is where integrations go to die.

I built a self-serve toggle so a merchant could mint their own test credentials from the production dashboard. The security question is obvious — you have just let people create credentials — so it is secured with a public/private key and DPoP (RFC 9449): a short-TTL proof-of-possession token bound to the HTTP method, the URL and a timestamp, verified at the gateway before auth even runs. The merchant id is embedded in the token, so a merchant can only ever mint their own test credentials.

Landing it meant design, product, integrations, infra, appsec and UI. It is the least glamorous list in this post and it was most of the work.

weeks → a day merchant integration time
~50% fewer integration issues

How we knew it was working

Every number in the section above came from somewhere, and building that somewhere was part of the job.

We instrumented the SDK with events, so we could see the integration funnel: where a merchant's engineer picked it up, where they got stuck, where they dropped, and where they finally put a live transaction through. A funnel turns "integration is painful" — which is an opinion — into a specific step that loses people, which is a thing you can go and fix.

On top of the events I built an internal dashboard with the analytics around it: weekly and monthly success rates, so adoption was something we watched rather than something we assumed. If a release made things worse, we would see it in a week instead of hearing it in a support ticket a month later.

An SDK you cannot measure is an SDK you are guessing about.

This is the part that gets skipped, and it is the part that made the rest possible. The self-serve test keys were not a hunch — the funnel showed us where merchants were stalling, and it was the credential step. You fix the step the data points at, and then you watch the same dashboard tell you whether it worked.

Four SDKs, one behaviour

I owned the SDK offerings, not just the Java one. The Python, .NET and JavaScript SDKs were built by three or four engineers — juniors and interns — and I mentored them rather than writing the SDKs myself.

The trap there is obvious in hindsight and invisible at the time: four SDKs written by four people become four dialects. Same gateway, four different ideas about what an error means, four retry policies, four ways of naming the same field. So I gated at the design level, not the code level. Get the behaviour agreed before anyone opens an editor, and the implementations converge on their own.

Making it public

Once the Java SDK was solid, it was living in an internal repository, serving only the merchants already in our sales funnel. That felt like a waste of the thing.

A public SDK is a channel. It reaches developers who will never talk to a salesperson, and it lets a merchant's engineers read the code before they commit to the payment gateway that will handle their money — which is exactly what a good engineer does before making that decision.

So I drove the open-source release: appsec approval, the architects' approvals, and the public repository. It is on GitHub at PhonePe/phonepe-pg-sdk-java.

That is the part I am most glad about, and not for the reasons I expected. It means that when I tell you I designed a payment SDK that merchants bet their checkout on, you do not have to take my word for it. You can go and read it.
